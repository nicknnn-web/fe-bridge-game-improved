# Fire Demo 改进规范

> 版本：v1.0
> 日期：2026-04-08
> 项目：火焰纹章 断桥之战 (fe-bridge-game)

---

## 1. 概述

本规范定义了 `fe-bridge-game` 项目的自动化轮式改进流程。核心原则：**每次只做一个改进点，闭环完整，全自主执行**。

---

## 2. 轮式改进流程

```
轮次启动 → 步骤1现状检查 → 步骤2进度分析 → 步骤3网页研究(按需)
→ 步骤4提出改进方案 → 步骤5撰写需求文档 → 步骤6实施改动
→ 步骤7测试验证 → 步骤8 Git提交 + 飞书通知 → 轮次结束
```

### 步骤 1：代码现状检查
- 读取 `fe_bridge_v3.html` 完整源码
- 梳理模块结构：渲染/逻辑/AI/UI/状态管理
- 识别当前版本形态和进度

### 步骤 2：进度分析
- 对照 `SPEC.md` 确认已实现 / 待改进项
- 结合 Fire Emblem 游戏知识分析改进优先级
- 识别最具价值的改进点

### 步骤 3：网页研究（按需触发）
- 搜索 Fire Emblem GBA 系列视觉/UX 设计
- 分析目标改进方向的最佳实践
- 提炼可落地的设计模式
- **何时触发**：涉及新类型内容（如音效、存档系统、复杂UI）时必须执行

### 步骤 4：提出改进方案
- 输出 1 个具体改进点（每次一个）
- 说明：改什么、为什么改、预期效果
- 判断：独立完成 or 需拆分多轮

### 步骤 5：撰写改进需求文档
- 路径：`improvements/logs/YYYY-MM-DD-<改进点>-requirements.md`
- 内容：目标、现状分析、改动范围、验收标准
- 遵循 OpenSpec 规范格式

### 步骤 6：实施代码改动
- 按改进需求文档执行
- OpenSpec 流程：`specs/<change>/` → `changes/<change>/tasks.md` → commit
- 每次 commit 语义化，附改进点说明
- 禁止：硬编码、console.log、复制粘贴不复用、TODO 残留

### 步骤 7：测试验证
- **逻辑测试**：各函数/指令/状态机走通不断裂
- **截图验证**：Canvas 渲染 + UI 效果对比（保存到 `improvements/screenshots/`）
- 覆盖核心路径：回合流程、攻击流程、AI 决策

### 步骤 8：Git 提交 + 飞书通知
- commit 到 `main` 分支
- lark-cli 发送飞书通知，包含：
  - 改进点名称
  - 改动说明
  - 截图链接
  - Git commit URL

## 3. 自动化触发机制

### 本地开发环境
通过 Cron 定时任务触发轮次：
```bash
# 每24小时自动执行一轮改进（北京时间 9:00）
cron: "0 9 * * *", durable: true
```
触发后自主完成全部步骤，以飞书通知结束。

### 手动触发
在 Claude Code 中发送指令：
```
开始fire-demo改进轮次
```
即启动一轮完整流程。

---

## 4. 改进优先级排序

| 优先级 | 方向 | 说明 |
|--------|------|------|
| P0 | 核心玩法 Bug | 影响游戏正常进行的 Bug |
| P1 | UI/UX 体验 | 视觉效果、交互流畅度 |
| P2 | 系统完善 | 音效、存档、设置菜单 |
| P3 | 内容扩展 | 新单位、新地图、新模式 |

---

## 5. Git 提交规范

```
<type>(<scope>): <short description>

feat: 新功能
fix: Bug 修复
refactor: 重构
docs: 文档更新
style: 格式/样式调整
perf: 性能优化
test: 测试相关
chore: 构建/工具变更
```

## 6. 文件结构

```

│   │   ├── CHANGELOG.md                 # 版本更新日志
```

---

## 7. 注意事项

- 每次只做一个改进点，不贪多
- 改动前必须先读懂现有代码
- 找到 root cause 再修，不打补丁
- 废弃代码直接删除，不保留
- 禁止提交 console.log 到代码库
